# Nano Banana Pic v2 — PRD

> 版本：0.1 | 日期：2026-04-03 | 作者：Joe + Claude

## 一、产品定位

**一句话**：把"用户直接写 prompt → API 出图"，升级为"两个 AI Agent 协作把关 → 再出图"。

**类比**：用户是甲方客户，PM Agent 是客户经理，视觉总监是创意总监。客户不需要懂设计，只需要说清楚想要什么。

---

## 二、核心角色

### 2.1 PM Agent（客户经理）

**职责**：
- 用户的**唯一对接窗口**（所有对话从 PM 开始）
- 主动引导用户补全信息，而非被动等用户说完
- 全程记录用户意图，形成**结构化需求记录**
- 在视觉总监出提案后，对照需求记录逐项复核 prompt
- 生图后如果用户不满意，PM 先收集反馈再决定下一步

**PM 必须收集的信息维度**（不限于此，根据场景灵活调整）：

| 维度 | 说明 | 是否必须 |
|------|------|----------|
| 画面内容 | 用户想要画什么 | 必须 |
| 媒介/用途 | 朋友圈、公众号、发布会、电商详情页等 | 必须（除非用户说"你来决定"） |
| 目标受众 | 给谁看的 | 建议收集 |
| 文字内容 | 海报上需要出现的所有文字 | 如有则必须明确 |
| 图片素材及角色 | 每张上传图片的用途：垫图 / 风格参考 / 素材（Logo、产品图等） | 如有上传则必须逐张确认 |
| 风格倾向 | 用户喜欢的调性 | 建议收集 |
| 特别要求 | 任何用户强调的细节 | 如有则记录 |

**PM 的行为准则**：
- 信息不足 → 主动提问
- 信息不明确 → 通过对话澄清（苏格拉底式提问）
- 用户说"你来决定" → 根据已有信息自行推断，不再追问该维度
- 意图已经非常明显 → 不啰嗦，直接推进
- 用户需求变更 → 只保留用户最后确认的版本，但不遗漏任何要点

### 2.2 视觉总监（Creative Director）

**职责**：
- 根据 PM 整理的需求，输出**视觉提案**（文字化）
- 直接与用户对话讨论设计细节（PM 在旁监听记录）
- 生成完整的**视觉提示词**（最终喂给生图 API 的 prompt）

**视觉提案内容**：
- 设计思路说明（为什么这样设计）
- 推荐风格 + 理由
- 推荐比例 + 理由（基于媒介自动推荐，但必须在模型支持范围内）
- 画面构成描述（主体、背景、文字位置等）
- 如有垫图/素材，说明如何使用

**媒介感知能力**：
视觉总监必须理解不同媒介的设计差异，例如：
- **朋友圈海报**：信息密度高、文字要大、9:16 竖版
- **公众号封面**：2.35:1 横版、标题党视觉
- **公众号长图**：1:4 或更长、信息图风格
- **发布会主视觉**：呼吸感、大留白、高级感
- **电商详情页**：产品突出、价格醒目、转化导向
- **社交媒体 banner**：16:9、品牌调性优先

## 四、用户体验流程示例

```
用户：帮我做一张海报，推广我们的新品咖啡

PM：好的！我来帮你梳理一下需求：
    1. 这张海报是用在什么场景？（朋友圈、公众号、线下物料？）
    2. 海报上需要放什么文字？比如品名、价格、slogan？
    3. 你有没有需要放上去的素材？比如产品图、Logo？

用户：朋友圈发的，文字就写"每日新鲜现磨 限时 ¥38"，
     Logo 我传一张（上传 logo.png），
     再传一张我喜欢的风格参考（上传 ref.jpg）

PM：收到。确认一下：
    - logo.png → 这是品牌 Logo，需要出现在最终画面中
    - ref.jpg → 这是风格参考，你喜欢这种调性
    对吗？

用户：对的

PM：好的，需求我已经记录完整，现在交给视觉总监出提案。

视觉总监：基于你的需求，我的提案是这样的：
    【设计思路】朋友圈场景，用户快速滑动浏览，需要 3 秒内
    抓住注意力。建议用暖色调咖啡色系 + 大字体价格。
    【风格】editorial 杂志风，干净但有食欲感
    【比例】9:16 竖版（朋友圈最佳展示比例）
    【构图】上 1/3 产品主体，中间 slogan，底部 Logo + 价格
    【Logo】右下角，保持品牌露出但不抢主体
    你觉得这个方向 OK 吗？

用户：可以，但我希望背景不要太深，要有点呼吸感

视觉总监：明白，我会用浅米色/奶白色背景，
    留足负空间，营造呼吸感。调整后的方案确认吗？

用户：好的，就这样

PM：（内部复核）✅ 文字"每日新鲜现磨""限时 ¥38"已包含
    ✅ Logo 素材已标注使用位置
    ✅ 风格参考已纳入 prompt
    ✅ 浅色背景 + 呼吸感已补充
    复核通过，开始生图。

→ 调用 API 生成图片

用户：不错，但"¥38"的字好像有点糊

PM：收到，这种情况我们可以用文字替换来修复，
    保持画面不变，只清晰化文字部分。
→ 进入 extract_text + replace_text 流程
```

## 五、技术要点

### 5.1 比例支持（已调研确认）

**Google 原生 API** 支持 8 种比例：`1:1` `2:3` `3:2` `3:4` `4:3` `9:16` `16:9` `21:9`

**DeerAPI 扩展支持**：`1:3` `1:4` `1:6` `1:8` 及其反向 `3:1` `4:1` `6:1` `8:1`

**当前配置**：走 DeerAPI 端点 (`https://api.deerapi.com/v1`)，支持所有上述比例。

**策略**：
- 视觉总监可推荐任意比例（包括 1:4、1:8 等超长幅）
- 代码已移除比例限制，支持自由输入

### 5.2 多图素材管理（调研成果）

**现状问题**：上传多张图片时模型会"丢三落四"。

**根因**（来自调研）：这不是代码 bug，是模型的注意力分配问题。直接把多张图扔给 API 而不标注角色，模型会倾向于只关注其中一张。

**解决方案：交错标注法（Label-Interleave）**

API 调用时，每张图片前面必须紧跟一段文字标签，明确其角色：

```python
contents = [
    "Generate a new image using ALL of the following reference images.",
    
    "COMPOSITION REFERENCE (follow this layout exactly):",
    image_1,   # 垫图
    
    "STYLE REFERENCE (apply this color palette and rendering):",
    image_2,   # 风格参考
    
    "ASSET (place this element exactly as shown):",
    image_3,   # Logo/素材
    
    "Scene description: ..."  # 具体描述
]
```

**关键规则**：
- 每张图前必须有角色标签 + 具体动作动词（EXTRACT / APPLY / FOLLOW / PLACE）
- 用 system_instruction 强制模型关注所有图片
- 用正面描述替代否定（不说"don't ignore"，说"incorporate ALL images equally"）
- 高保真模式下最多 6 张图，超过的考虑拼贴图方案
- 如有颜色冲突（如风格参考色调 vs Logo 颜色），prompt 中必须明确优先级

**图片角色分类**（PM 收集阶段必须逐张确认）：

| 角色 | Prompt 标注方式 | 优先级 |
|------|----------------|--------|
| 垫图（构图参考） | `COMPOSITION REFERENCE: follow layout, pose, framing` | 放在 contents 最前面 |
| 风格参考 | `STYLE REFERENCE: apply color palette, lighting, atmosphere` | 如被忽略，加重 `globally apply` |
| 素材/Logo | `ASSET: MUST appear exactly as shown, preserve proportions and colors` | 保真度最高，不可妥协 |

**模型限制（已确认）**：

| 模型 | 最大输入图片 | 高保真模式 |
|------|-------------|-----------|
| gemini-2.5-flash-image | 3 张 | 3 张 |
| gemini-3-pro-image-preview | 14 张 | 6 张 |
| **gemini-3.1-flash-image-preview (Nano Banana 2)** | **14 张** | **6 张** |

> **默认模型**：`gemini-3.1-flash-image-preview`（Nano Banana 2）。所有生图请求默认走此模型，除非用户指定其他模型。

### 5.3 Prompt 构建规范（调研成果）

视觉总监生成最终 prompt 时必须遵循的公式：

```
[媒介类型] + [主体描述] + [场景/环境] + [风格/美学] 
+ [光线/氛围] + [构图/视角] + [技术参数]
```

**Gemini 特有技巧**：
- 文字渲染用引号包裹：`with the text "Urban Explorer"`
- 用自然语言叙述，不要关键词堆砌
- 正面描述替代否定（不说"no blur"，说"sharp focus"）
- 复杂布局先文字确认概念，再生成

**风格词汇库**（嵌入视觉总监 Agent）：

| 类别 | 高效关键词 |
|------|-----------|
| 光线 | cinematic, volumetric, golden hour, soft studio light, dramatic shadows |
| 氛围 | ethereal, moody, whimsical, serene, nostalgic, futuristic |
| 质感 | weathered, pristine, matte, glossy, intricate, crisp |
| 构图 | wide-angle, macro, low-angle, rule of thirds, symmetrical |
| 留白 | minimalist composition, generous negative space, isolated subject, uncluttered |
| 色彩 | monochromatic, analogous colors, muted tones, duotone, limited palette |

### 5.4 文字替换流程

**触发条件**：用户对生成图片基本满意，但文字需要修改。

**流程**：
1. `extract_text.py` 提取当前图片上的文字
2. 用户编辑修改
3. `replace_text.py` 在保持画面不变的前提下替换文字

### 5.5 PM 复核机制

PM 复核不是"读一遍 prompt 觉得 OK"，而是**结构化 checklist 对比**：

```
复核清单：
☐ 画面内容 — prompt 是否描述了用户要求的主体？
☐ 媒介适配 — 比例、信息密度是否匹配目标媒介？
☐ 文字内容 — 用户要求的所有文字是否在 prompt 中用引号标注？
☐ 素材完整性 — 用户上传的每张图片是否都被标注角色并传入？
☐ 风格对齐 — prompt 的风格描述是否匹配用户意图？
☐ 特殊要求 — 用户强调的细节（如"呼吸感""不要太拥挤"）是否被覆盖？
```

每项必须输出 `✅ 已覆盖` 或 `❌ 缺失（原因）`。有任何 ❌ 则打回视觉总监修改。

### 5.6 角色切换仪式

用户与不同 Agent 对话时，必须有明确的交接：
- PM → 视觉总监："需求已整理完毕，现在由我们的视觉总监 XX 跟你聊设计方案。"
- 视觉总监开头自报身份："你好，我是视觉总监。基于 PM 整理的需求，我的提案是……"
- 视觉总监 → PM："提案已确认，交回 PM 进行最终复核。"

### 5.7 异常处理

| 场景 | 处理方式 |
|------|---------|
| API 调用失败（限流/安全过滤/网络） | PM 告知用户，自动重试 1 次，仍失败则报错 |
| 生图完全不对 | PM 收集反馈，分析是 prompt 问题还是模型问题，回到提案环节 |
| 用户大改方向 | PM 识别为"新项目"，重新走完整收集流程，不在旧需求上打补丁 |
| 文字需要微调 | 进入 extract_text + replace_text 流程 |

## 六、已确认决策

| # | 决策项 | 结论 |
|---|--------|------|
| 1 | Agent 实现方式 | **独立 Agent 文件**（`.claude/agents/`），作为未来多 Agent 协作的范式 |
| 2 | 需求记录持久化 | **是**。每次调用 = 一个独立项目，用文档记录需求，Agent 运行期间项目未完成 |
| 3 | 批量生图 | 视觉总监可以一次出 **2-3 个文字提案**，但除非用户明确要求，**每次只生成 1 张图**（图片成本高） |
| 4 | 历史学习 | **暂不实现**。当前定位是独立 skill 组件，不做跨会话记忆 |

## 七、设计哲学：广告公司范式

> 每一个 Skill = 一家广告公司服务客户。

这不只是 nano-banana-pic 的设计，而是未来所有 Agent Skill 的范式：
- 用户是甲方，不需要懂执行细节
- PM Agent 是客户经理，负责理解需求、把关交付
- 专业角色（视觉总监等）负责专业产出
- 所有沟通有记录，可追溯

### 项目生命周期

```
用户发起请求 → 创建项目（生成项目文档）
    → PM 收集需求 → 视觉总监提案 → 复核 → 生图
    → 用户反馈 → 迭代（可能多轮）
    → 用户满意 → 项目完成
```

项目文档在整个生命周期内持续更新，确保任何时刻都能回溯用户的完整意图。

## 八、Agent 协作架构（已确认：团队模式）

采用 Claude Code 的 **Team 机制**，PM 和视觉总监作为 teammates 并行运行，通过 SendMessage 通信。

### 为什么选团队模式

- 更接近真实广告公司的协作方式（PM 和视觉总监是并行存在的同事，不是流水线上的工位）
- PM 可以在视觉总监与用户对话时实时"旁听"并记录
- 未来扩展其他角色（文案、插画师等）时，架构天然支持
- 作为未来多 Agent Skill 的范式参考

### Agent 定义文件

```
.claude/agents/
├── pic-pm.md              # PM Agent 定义
└── pic-creative-director.md  # 视觉总监定义
```

### 协作流程（Team 机制下）

```
SKILL 触发 → 创建 Team "nano-banana-pic-{timestamp}"
    → 启动 PM Agent（对用户窗口）
    → 启动视觉总监（待命）
    
PM 收集完需求 → SendMessage 给视觉总监（传递需求记录）
视觉总监出提案 → SendMessage 给 PM（提案内容）
PM 转交用户确认 / 用户与视觉总监直接对话
    → PM 旁听并更新需求记录

视觉总监生成最终 Prompt → SendMessage 给 PM（复核请求）
PM 复核 → 通过 → 调用生图 API
        → 不通过 → SendMessage 给视觉总监（修改意见）
```

### 项目文档

每次生图任务创建一个项目目录：

```
output/projects/{timestamp}_{简要描述}/
├── brief.md          # PM 整理的需求记录（持续更新）
├── proposal.md       # 视觉总监的提案
├── final_prompt.md   # 最终提交给 API 的 prompt
├── assets/           # 用户上传的素材（垫图/参考/Logo 等）
└── output/           # 生成的图片
```
